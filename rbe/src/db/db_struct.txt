This is a very common backend design problem — and you’re thinking in the right direction: auth data separated from profile data + user roles/levels.

Let’s design it cleanly so it scales and doesn’t become painful later.

🧠 Core idea (important)

You actually have 3 different concerns:

Authentication (who are you?)
Authorization (what can you do?)
Profile data (what do we know about you?)

Don’t mix them.

🧱 Recommended schema (clean + scalable)
1. users (AUTH ONLY)

This table should stay minimal:

CREATE TABLE users (
    id UUID PRIMARY KEY,
    email TEXT UNIQUE NOT NULL,
    password_hash TEXT NOT NULL,
    created_at TIMESTAMPTZ NOT NULL DEFAULT now()
);

👉 This is ONLY login identity.

2. user_profiles (ONE-TO-ONE)
CREATE TABLE user_profiles (
    user_id UUID PRIMARY KEY REFERENCES users(id) ON DELETE CASCADE,
    display_name TEXT,
    avatar_url TEXT,
    bio TEXT
);

👉 1 user → 1 profile
👉 optional fields, safe to expand later

3. user_roles (MANY-TO-MANY, recommended)

Instead of “guest/coach/partner column”, do this:

CREATE TABLE roles (
    id SERIAL PRIMARY KEY,
    name TEXT UNIQUE NOT NULL
);
CREATE TABLE user_roles (
    user_id UUID REFERENCES users(id) ON DELETE CASCADE,
    role_id INT REFERENCES roles(id) ON DELETE CASCADE,
    PRIMARY KEY (user_id, role_id)
);
⚠️ Why NOT store user_type in users?

You might think:

users: id, email, password, type

❌ This breaks quickly because:

users can have multiple roles (coach + partner)
roles evolve over time
permissions become messy
migrations become painful
🧩 How your system works now
Example:

User: John

Table	Data
users	email, password
user_profiles	display_name = "John"
user_roles	coach + partner
🔐 Authorization model (important)

Instead of:

if user.type == "coach"

You do:

if user has role "coach"

Much more flexible.

🦀 Rust + SQLx modeling idea
User
struct User {
    id: Uuid,
    email: String,
    password_hash: String,
}
Profile (1:1)
struct UserProfile {
    user_id: Uuid,
    display_name: Option<String>,
    avatar_url: Option<String>,
    bio: Option<String>,
}
Role system
struct Role {
    id: i32,
    name: String,
}
⚡ Query pattern (SQLx style)
Get user + roles
SELECT r.name
FROM roles r
JOIN user_roles ur ON ur.role_id = r.id
WHERE ur.user_id = $1;
Get full user “view”

You typically assemble it in service layer:

user
profile
roles

👉 NOT one giant SQL query (unless needed)
